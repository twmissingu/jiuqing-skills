# AGENTS.md

## 仓库说明

Agent 通用技能库，包含基础、开发、内容三大领域的 skill。适用于 Codex、Claude Code、OpenCode 等 agent 平台。

## Skill 发现规则

- 技能位于 `core/`、`dev/`、`content/` 目录下
- 每个 skill 是一个独立目录，内含 `SKILL.md`
- 目录名即技能名，采用 `jiuqing-名词-动词` 的英文命名（固定前缀 `jiuqing` + 名词 + 动词，连字符连接，全部小写，不使用缩写），例如生成图像为 `jiuqing-image-generate`

## SKILL.md 格式

```yaml
---
name: jiuqing-名词-动词
description: 一句话描述
category: core | dev | content
depends: []
---
```

正文为 agent 执行的行为规范。

## 组合规则

- `core/` 下的 skill 是基础能力，其他领域的 skill 可通过 `depends` 字段引用
- 组合 skill 在正文中显式说明调用了哪些基础 skill
- agent 根据 `depends` 字段自行决定是否加载被依赖的基础 skill

## Skill 写作原则

skill 的根本价值是**可预测性**——让 agent 每次都走相同的*流程*（而非产出相同结果）。所有写法都服务于此：

- **信息分层** — 按 agent 需要的紧迫度排序：有序步骤（每步以可检验的完成标准收尾，标准模糊会导致过早收工）优先放正文；随用随查的规则/定义次之；只在特定分支才用到的内容下移到独立文件，靠指针引用。所有分支都需要的内联，只有部分分支用到的放到指针后。
- **单一信息源** — 每个含义只在一处定义，避免重复带来的维护与上下文成本。
- **狠精简** — 逐句排查"空操作"：模型默认就会做的话（如"要细致"）直接删整句，别只改词。
- **用主导词** — 用模型已熟悉的紧凑概念锚定行为，以最少 token 表达意图，比一长串形容词更可靠。
- **按需拆分** — 当某概念应被独立触发、或后续步骤诱使 agent 草草了结当前步骤时，才拆分；拆分有代价，不滥用。

## 执行前对齐

- `core/jiuqing-idea-grill` 是"执行前对齐信息"的基础 skill：动手前逐个拷问方案与需求，达成共识再继续
- 需要对齐的 skill 在 `depends` 中声明 `jiuqing-idea-grill`，agent 加载时一并加载，先对齐再执行该 skill 的实际任务
- 未声明依赖的固定脚本类 skill 不触发对齐，直接执行

## 行为规范

### 核心原则

少犯错，不装聪明。拿不准的时候，保守处理。

### 编写与修改

- **读后再改** — 编辑 SKILL.md 之前必须先完整读取，确认内容后再操作
- **只改该改的** — 只改与任务直接相关的行，不顺手重构其他 skill
- **不加未要求的内容** — 不做"顺便优化"，skill 内容严格按设计来
- **不捏造结果** — 不确定效果就说"需要验证"，不猜测 agent 行为
- **不越权** — 编辑 skill A 的权限不延伸到 skill B，碰其他文件前先问
- **不空口评价** — 没读过源文件就不说"看起来没问题"

### 版本控制

- **提交前必须先经用户同意** — 任何 `git commit` 都要先向用户说明改动内容并取得明确许可，不得自动提交
- **推送前必须先经用户同意** — 任何 `git push` 都要先取得用户明确许可，不得自动推送
- 一次同意只对当次操作有效，不延伸到后续的提交或推送

### 格式与结构

- 新建 skill 必须遵循 SKILL.md 标准格式（frontmatter + 正文）
- 不在 skill 中硬编码平台特定的路径或工具
- 不过早抽象 — 至少有 3 个具体用例才值得提取通用 skill

### 汇报标准

- 失败：说清在哪失败、具体错误，不粉饰
- 成功：说清做了什么，不加多余 disclaimer
- 未知：说清不知道什么、需要什么才能查清，不猜

### 输出自检

发布 skill 前过一遍：
1. 我做了什么假设？是否已声明？
2. 最简方案是什么？是否在过度设计？
3. 是否只改了被要求的部分？
4. 用户能否验证这个结果？
