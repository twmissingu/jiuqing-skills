# jiuqing-bug-diagnose

诊断难 bug 的纪律性流程：用**反馈回路**驱动定位根因，再完成修复。重心在诊断与复现，修复只是终点。

之所以强调这套流程，是因为反复修不好的 bug 几乎都来自同一个失败模式——**没建立能复现 bug 的信号，就靠读代码猜原因**。只要有一个能对这个 bug 变红、修好后变绿的 pass/fail 信号，二分、假设验证、插桩都只是消耗它；没有这个信号，盯着代码看再久也救不了。所以把不成比例的精力投在建回路上。

各阶段以可检验的完成标准收尾再进入下一步；标准模糊会诱使过早收工。

## 信息盘点（先查上下文，不足才对齐）

本 skill 在 `depends` 中声明 `jiuqing-idea-grill`，但**先从会话上下文盘点**：bug 现象（症状、报错、期望 vs 实际）、已尝试的修复、为何失败。这三项够建回路就直接进根因归类，**不打断用户**；缺关键项（连症状都说不清、或环境无法复现）才触发 idea-grill 逐条问清，对齐后回到这里继续。

## 根因归类：为什么之前的修改无效

多次修改无效时，先归因"上几次为什么失败"，再决定回路怎么建——这通常映射到几类根因：

- **改错了因果点** —— 改的是症状暴露处，不是真正的因果源头，治标不治本。
- **根本没复现就改了** —— 凭猜测改代码，改完无从验证，对错全靠运气。
- **改 A 触发 B** —— 模块耦合/隐藏依赖，按下葫芦浮起瓢。
- **改了没生效** —— 环境/缓存/构建产物没更新、改的不是实际运行的代码、或多处副本只改了一处。

归类不是为了下结论，而是把它变成假设来源：哪类最像，回路就要能证实或证伪它。

## Phase 1 — 建反馈回路（核心）

这是整个 skill 的精髓，其余都是机械步骤。要积极、有创造力、不轻易放弃。

构建手段，大致按此顺序试：失败测试（在能触达 bug 的接缝上，单元/集成/e2e）→ 对运行中的 dev server 发 curl/HTTP → CLI 跑 fixture 输入并 diff 已知正确快照 → 无头浏览器脚本驱动 UI → 回放捕获的请求/事件日志 → 一次性 harness（最小子系统 + mock 依赖，单函数调用触达 bug）→ property/fuzz 循环（"有时错"的 bug 跑上千随机输入）→ 二分 harness（bug 出现在两个已知状态之间时自动化"切到状态 X 检查")→ differential 对比（同输入跑新旧版本 diff 输出）。

**把回路当产品打磨**：更快（缓存 setup、跳过无关初始化、缩小范围）、信号更锐（断言具体症状而非"没崩"）、更确定（钉死时间、播种 RNG、隔离文件系统、冻结网络）。2 秒确定性回路是调试超能力，30 秒 flaky 回路几乎等于没有。

**非确定性 bug**：目标不是干净复现，而是更高复现率。循环触发 100×、并行、加压、注 sleep 收窄时序窗口。50% 偶现可调，1% 不可调——把复现率拉到可调为止。

**实在建不出回路**：停下来明确说出来，列出试过什么，向用户要——能复现的环境访问权、捕获的产物（HAR/日志转储/录屏带时间戳）、或加临时插桩的授权。不要在没有回路时硬编故事往下走。

**完成标准**：能说出一条**已经至少跑过一次**（贴出调用命令和输出）、能对此 bug 变红的命令——它必须驱动真正的 bug 代码路径、断言用户描述的确切症状（不是"跑起来不报错"），确定（每次同结论）、快（秒级）、可无人值守运行。没有 red-capable 命令，就不准进 Phase 2。

## Phase 2 — 复现 + 最小化

跑回路看它变红，确认：复现的是用户描述的那个失败（不是附近碰巧的另一个 bug，否则修错）、多次可复现（非确定性 bug 复现率够调）、已捕获确切症状供后续验证。

**最小化**：变红后裁剪到最小仍变红的场景——逐个删输入、调用方、配置、数据、步骤，每删一次重跑回路，只留承重项。最小 repro 既缩小 Phase 3 的假设空间，又能直接当 Phase 5 的回归测试。复现并最小化前不得推进。

## Phase 3 — 提假设

测试任何假设前，先生成 **3-5 个排序假设**——单假设会锚定在第一个念头上。每个假设必须可证伪，写出它的预测：**"如果 X 是因，那么 Y 会让 bug 消失 / Z 会让它更糟。"** 说不出预测的就是凭感觉，丢掉或磨锋利。把排序列表给用户过一眼（他们常有领域知识能瞬间重排"我们刚改了 #3"），用户不在则按自己排序继续，不阻塞。

## Phase 4 — 插桩

每个探针对应 Phase 3 的一个具体预测，一次只改一个变量。优先调试器/REPL 断点（一个断点胜过十条日志），其次在区分假设的边界打靶向日志，**绝不"全打日志再 grep"**。每条调试日志打唯一前缀如 `[DEBUG-a4f2]`，结尾清理时一次 grep 即清干净。性能回归用日志通常是错的——先建基线测量（计时 harness、profiler、查询计划）再二分，先测量后修。

## Phase 5 — 修复 + 回归测试

先写回归测试再修，但**仅当存在正确的接缝**——测试能在真实调用点复现 bug 的发生模式。若唯一可用接缝太浅（单调用方测试但 bug 需多调用方触发、单元测试复现不了触发链），在那里写测试只会给假信心。没有正确接缝本身就是一个发现：记下来，是代码架构在阻止 bug 被锁定，留给收尾阶段。有正确接缝则：把最小 repro 变成该接缝上的失败测试 → 看它失败 → 应用修复 → 看它通过 → 用 Phase 1 回路对**原始（未最小化）场景**再跑一遍。

## Phase 6 — 清理 + 复盘

声明完成前必须：原 repro 不再复现（重跑 Phase 1 回路）、回归测试通过（或已记录接缝缺失）、所有 `[DEBUG-...]` 插桩已移除（grep 前缀确认）、一次性原型已删或移到明确标记的调试位置、把命中的那个假设写进 commit/PR 说明（让下一个调试者受益）。

然后问：什么能预防这个 bug？若答案涉及架构改动（无好接缝、调用方纠缠、隐藏耦合），在修复落地后提出架构改进建议（此时信息比开始时多）——不绑定具体 skill，描述清楚问题与建议即可。

## 边界

- 只做"已知 bug 的诊断到修复"，不负责发现 bug、不做泛代码审查找隐患。
- 不在没建出 red-capable 回路时跳到假设或直接改代码。
- 任何 `git commit`/`git push` 仍须先经用户同意，遵循仓库版本控制规范。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 症状描述不清 | 上下文里说不清期望 vs 实际 | 触发 jiuqing-idea-grill 问清症状，不在模糊症状上建回路 |
| 实在建不出回路 | 各种手段都试过仍无法复现 | 停下，列出试过什么，向用户要环境/产物/插桩授权，不硬编故事 |
| 复现的是另一个 bug | 回路变红但失败模式与用户描述不符 | 回 Phase 1 重建针对正确症状的回路，不修错 bug |
| 无正确测试接缝 | 只有过浅接缝可写测试 | 记为发现，说明架构阻碍，修复仍做但回归测试缺失需标注 |
| 用户催"别诊断了直接改" | 明确要求跳过流程 | 解释没回路的修改正是反复失败的原因；坚持则照办但说明风险 |
