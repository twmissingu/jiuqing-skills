# jiuqing-product-polish

让 agent 自主循环迭代一个 vibe coding 项目，自动发现问题、修复打磨，直到逼近可交付——把"一次性生成代码"升级为"持续改进到能用"。

核心机制是**单维度聚焦的迭代循环**：每轮先给项目各维度打分，锁定**一个最弱维度**做最小改动，改动必须过**代码审核关卡**才落地（不过则回滚本轮），然后记日志、检测是否收敛。之所以一次只攻一个维度，是因为大而全的改动既难审核也难归因——小步快跑、每步可验证，才能稳定累积质量而不引入回归。

## 执行前对齐（仅此一次）

本 skill 在 `depends` 中声明 `jiuqing-idea-grill`：进入循环**之前**先走一遍对齐流程，一次只问一个、带推荐答案、能查项目就别问。这是全程唯一的人工介入点——对齐之后进入全自动循环，不再逐轮打断用户。所以对齐要把循环的边界一次问清。

对齐要点：

- **可交付标准** —— 这个项目的"打磨到可交付"终点画在哪？(核心流程端到端跑通？构建+测试全绿？达到某质量线？) 这决定收敛判定的目标。
- **迭代范围与红线** —— 哪些部分可以动、哪些必须保留不动、哪些行为不能破坏。
- **轮次上限** —— 最多迭代多少轮(默认 10)，防止无限循环空耗。
- **约束** —— 技术栈限制、不能引入的依赖、必须遵守的现有风格。

对齐后用一两句话复述共识，再进入 Phase 0。

## Phase 0 — 项目检测与初始化

1. **检测项目类型** —— 任一信号存在即为已有项目：包管理文件(`package.json`/`pyproject.toml`/`Cargo.toml`/`go.mod`/`pom.xml`/`*.uproject`/`CMakeLists.txt`/`Package.swift`)、源码目录(`src/`/`app/`/`lib/`/`Source/`)、≥3 个代码文件、`README.md`/`.git/`。据此判断技术栈与项目类别(Web/后端/移动/桌面/游戏等)。
2. **断点续传** —— 若已存在 `docs/.polish-checkpoint.json`，按 `convergence.md` 恢复上次进度，从中断处继续，不重头来。
3. **理解与文档初始化** —— 已有项目先读包管理文件、目录结构、1-2 个核心文件摸清风格与约定。在 `docs/` 下建立(或更新)文档体系：`VISION.md`(愿景，推断不出标"待确认")、`ROADMAP.md`(已实现 + 明显缺失)、`POLISH_LOG.md`(Round 0 基线)、`ARCHITECTURE.md`(当前架构)。
4. **确定评估维度** —— 读 `dimensions.md`，按项目类型选定本项目的评估维度集。

## 迭代循环（每轮 7 步）

循环条件：`当前轮次 < 轮次上限 且 未收敛`。每步以可检验的结论收尾再进入下一步，标准模糊会导致过早收工。

1. **上下文快照** —— 生成轻量摘要(技术栈、当前轮次、`git diff --name-only` 最近改动、核心模块入口与签名、构建/测试命令)，供本轮聚焦用，**不堆大段源码**。
2. **维度评估** —— 按 `dimensions.md` 给各维度打分(1-10，每分附具体证据)，按其中的选择规则挑出**本轮聚焦的那一个维度**及改进方案(新增功能 ≤2 个)。
3. **终端用户视角体验** —— 以首次使用者视角体验产品：Web 项目能启动就实测交互，否则转代码走读(引导清晰度、核心流程、错误反馈、边界情况)，列出最关键的痛点喂给开发步骤。
4. **最小改动开发 + 写测试** —— 只实现聚焦维度的改进，遵循现有风格、改动最小化，**不顺手扩张到别的维度**。为改进写测试，覆盖核心 + 边界(正向 + 反向)，运行确认通过。
5. **审核关卡(过关才落地)** —— 对本轮 `git diff` 做代码审核：质量、安全、架构、过度设计、改动范围、接口兼容性。有阻断问题就修复后重审，最多 3 次；3 次仍不过则**回滚本轮**(用 `git checkout HEAD -- <文件>` 还原代码、`rm` 删新增未追踪文件、**禁用 `git reset --hard`** 以保留轨迹，`docs/` 文档保留)，并在日志记为"已回滚"。
6. **安全审查(条件触发)** —— 本轮改动涉及认证/权限/数据处理时，扫描 secret 泄露、注入、认证缺陷并修复。
7. **记录与收敛检测** —— 追写 `POLISH_LOG.md`(本轮聚焦维度、改进、各维度评分、痛点、完成/回滚状态、关键变更)、更新 `ROADMAP.md`、保存 `docs/.polish-checkpoint.json`，再按 `convergence.md` 的决策树判断是否继续。继续则回到第 1 步。

各步骤的失败兜底策略见 `convergence.md` 的错误处理表。

## 交付收尾

收敛或达到轮次上限后：

1. **最终构建验证** —— 跑构建/测试确认无错(游戏引擎项目构建验证可选，见 `dimensions.md`)，失败则修复重试(≤3 次)。
2. **ROADMAP 收口** —— "进行中"→"已完成"，未做项移入"后续迭代"，补一段当前状态总结。
3. **交付报告** —— 读 LOG/ROADMAP/VISION 生成 `docs/DELIVERY_REPORT.md`：项目概况、总轮次、稳定功能、已知限制、各维度最终评分、后续建议。
4. **跨文档一致性审计** —— 核对 DELIVERY_REPORT / ROADMAP / POLISH_LOG 的事实、评分、状态是否一致，有矛盾以 POLISH_LOG 为准修正。
5. **清理** —— 删除临时文件(`docs/.polish-checkpoint.json`、上下文快照)。
6. 提示用户：打磨已到可交付状态，如需正式发版可用 `jiuqing-project-ship` 走发版前把关。

## 边界

- 只做"迭代打磨到可交付"，**不做对外发版动作**(打 tag、推送、创建 Release 交给 `jiuqing-project-ship`)。
- 不擅自扩大对齐时定下的迭代范围，不动声明保留的部分。
- 回滚用的 `git checkout`/`rm` 属本地恢复操作；任何 `git commit`/`git push` 仍须先经用户同意，遵循仓库版本控制规范。
