# 评估维度与维度选择

SKILL.md 的「Phase 0」与迭代循环「第 2 步」引用本文件。用于确定项目按什么维度评估、每轮聚焦哪个维度。

## 按项目类型的维度集

先判断项目类别，再取对应维度。判断不确定时，从 Web / 非 Web 两类里选更贴近的一类。

| 项目类型 | 评估维度 |
|----------|----------|
| Web 应用 | 用户体验、UI 精细度、功能完整性、性能、可靠性、可访问性 |
| 后端 / API | 功能完整性、可靠性、性能、安全性、代码质量、测试覆盖 |
| 移动端 | 用户体验、UI 精细度、功能完整性、性能、可靠性、测试覆盖 |
| 桌面应用 | 用户体验、UI 精细度、功能完整性、性能、可靠性、测试覆盖 |
| 游戏 | 玩法完整性、视觉与音效、性能、可靠性、代码质量、测试覆盖 |
| 非 Web(库 / CLI / 工具等) | 功能完整性、可靠性、性能、代码质量、测试覆盖、构建与部署 |

若对齐时用户给了自定义维度，以用户的为准。

## 游戏引擎项目的特例

检测到 `*.uproject`(UE5)、`Assets/*.unity`(Unity)、`project.godot`(Godot)时：维度固定用「游戏」一行；**构建验证降级为可选**——这类项目的构建依赖引擎 GUI，命令行常跑不通，构建失败不作为阻断，改以代码走读 + 逻辑检查替代。

## 每轮的维度选择规则

每轮只聚焦**一个**维度。按下面顺序挑：

1. **优先得分 < 7 的维度** —— 这些是离可交付最远的硬伤，先补。
2. **在候选里选最低分** —— 同为短板时，最弱的优先。
3. **优先未迭代过的** —— 参考 checkpoint 里的 `previousDimensions`，避免反复磨同一维度而冷落其他。
4. **避开连续失败的维度** —— 参考 `failedDimensions`(连续 2 轮回滚/无改进的维度)，它们短期难推进，跳过去先做能动的。
5. **同分时选"用户感知提升最大"的** —— 几个维度评分接近时，挑改进后用户最直接受益的那个。

若所有维度都 ≥ 7，仍选其中最低分的继续微调，直到收敛检测判定终止(见 `convergence.md`)。

## 打分要求

- 每个维度给 1-10 分，**必须附具体证据**(指出是哪个文件/哪段逻辑/哪个缺失支撑这个分)，不凭印象空给。
- 读不到、无法判断的维度如实标注，不为凑齐而编造证据。
