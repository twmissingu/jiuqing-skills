# 收敛检测、断点续传与错误处理

SKILL.md 的「Phase 0」(断点续传)与迭代循环「第 7 步」(收敛检测)引用本文件。

## 收敛检测决策树

每轮第 7 步记完日志后，按此树判断是否进入下一轮。设计目标是：**有改进空间就继续，确认推不动了就果断停**，避免无限循环空耗。

```
连续回滚/无改进 ≥ 2 轮？(consecutiveSkips >= 2)
├── 是 → 终止(推不动了)
└── 否 → 所有维度都 ≥ 7/10？
    ├── 是 → 还有显著改进 或 有未完成的核心(P0/P1)任务？
    │   ├── 是 → 继续(选最低分维度微调)
    │   └── 否 → 终止(已收敛到可交付)
    └── 否 → 存在 < 7 分且未迭代过的维度？
        ├── 是 → 继续(按 dimensions.md 选最弱维度)
        └── 否 → 上一轮无显著改进？
            ├── 是 → 终止
            └── 否 → 继续
```

判定用语：

- **显著改进** —— 本轮任一维度评分较上轮提升 ≥ 1 分(或解决了一个明确痛点)。微小波动不算。
- **达到轮次上限** —— 任何时候 `当前轮次 >= 轮次上限`(对齐时定的)都立即终止，优先级高于上表。

## 断点续传(checkpoint)

每轮第 7 步把进度写入 `docs/.polish-checkpoint.json`（普通文件，不需要任何 runtime 支持）；Phase 0 检测到该文件就恢复进度、从中断处继续。

字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `iteration` | number | 已完成的迭代轮次 |
| `consecutiveSkips` | number | 连续回滚/无改进的轮数(收敛检测用) |
| `previousDimensions` | string[] | 已聚焦过的维度名(避免重复磨同一维度) |
| `failedDimensions` | string[] | 连续 2 轮回滚/无改进的维度(短期跳过) |
| `dimensionHistory` | Record<string, number[]> | 每个维度的历史评分序列(判断改进趋势) |
| `lastFocus` | string | 上轮聚焦的维度 |

恢复时：读出 `iteration` 接着往下数，用 `previousDimensions`/`failedDimensions`/`dimensionHistory` 还原维度选择上下文，不重头评估已稳定的维度。

## 错误处理表

迭代循环中各步骤失败时的兜底策略——目标是**单步失败不拖垮整个循环**，能降级就降级、能跳过本轮就跳过，只有审核关卡失败才回滚。

| 步骤 | 失败策略 | 循环是否继续 |
|------|---------|-------------|
| 项目检测 | 用默认值(按"非 Web"维度集) | 继续 |
| 维度评估 | 跳过本轮，`consecutiveSkips` +1 | 进入下一轮 |
| 用户视角体验 | 用代码走读得出的默认痛点列表 | 继续 |
| 开发 / 写测试 | 当作本轮失败，进入审核关卡判定 | 视审核结果 |
| 审核关卡(3 次未过) | 回滚本轮(`git checkout` 还原、`rm` 删新增、保留 docs、禁用 `reset --hard`)，日志记"已回滚"，`consecutiveSkips` +1 | 回滚后继续 |
| 安全审查 | 修复发现的问题后重新验证 | 修复后继续 |
| 最终构建验证 | 修复重试 ≤3 次 | 仍失败则如实记入交付报告 |

任何步骤抛出的异常都要如实记录到 `POLISH_LOG.md`，不静默吞掉、不粉饰成"通过"。
