# jiuqing-prd-write

把**当前对话已经讨论清楚的内容**综合成一份结构化 PRD（产品需求文档），输出为本地 Markdown 文件。本 skill 只综合已有讨论，不从头采访；只产出文档，不写代码。从零理清想法是 `jiuqing-idea-grill` 的事。

## ① 判断上下文是否充足

先评估当前对话和代码库现状，是否已经足够写出一份 PRD。判断依据：要解决的问题、目标用户、核心方案、关键取舍是否都已在对话里明确。

- **充足** —— 直接进入 ②（探查代码库补全细节），跳过对齐。这是 to-prd 式的主路径：只综合，不访谈。
- **不足** —— 关键信息缺失（如问题本身没说清、方案有多个未定分支），先走 ③ 的对齐流程补齐，再回到 ② 继续。

判断结果用一句话告知用户（"对话信息已足够，我直接综合"或"有 X 处还没定，先对齐一下"），避免用户困惑于你为什么问或不问。

## ② 探查代码库 + 划定测试接缝

写 PRD 之前先摸清代码库现状，让需求落到真实的代码上：

1. **探查现状** —— 读相关模块、目录结构、已有约定，搞清这个功能要落在哪、和现有代码怎么衔接。沿用项目既有的领域术语写 PRD，尊重相关 ADR/设计约定。
2. **划定测试接缝（seams）** —— seam 是测试这个功能时切入的边界。原则：
   - **复用优先**：先找代码库里已有的同类测试（如 integration/、e2e/），如果已有合适的 seam 就直接复用，不新建。
   - **高位切入**：seam 应设在被测功能的最高公共入口（如 Service 层而非具体方法），这样测试能覆盖最多代码路径。
   - **数量控制**：seam 总数不超过 3 个；如果能用 1 个 seam 覆盖主流程，就不要拆成多个。
   - 划完用**一句话列出 seam 名称和位置**和用户确认是否符合预期——这是主路径上唯一的人工确认点。接缝选错会让后续测试设计整体跑偏，而它又不像功能需求那样在对话里被自然讨论过。

## ③ 对齐补齐（仅在 ① 判定不足时触发）

本 skill 在 `depends` 中声明 `jiuqing-idea-grill`：仅当上下文不足时，在产出 PRD **之前**走一遍对齐流程，把缺失的关键信息逐条问清楚。一次只问一个问题，每个问题带上你的推荐答案，能从对话或代码读到的就别问。

对齐要点（只问 ① 中判定缺失的那几项，不全问）：

- **问题** —— 用户真正面对的问题是什么？从用户视角。
- **方案** —— 准备怎么解决？从用户视角。
- **范围** —— 哪些做、哪些明确不做。
- **取舍** —— 有没有未定的架构/接口/schema 分支需要拍板。

对齐后用一两句话复述补齐的共识，再回到 ②。

## ④ 产出 PRD

按下面七段模板写，输出为本地 Markdown 文件（默认 `docs/PRD-<功能名>.md`，目录不存在则先建；与用户另有约定则从其约定）。

```markdown
## 问题陈述（Problem Statement）
用户正面对的问题，从用户视角写。

## 解决方案（Solution）
解决该问题的方案，从用户视角写。

## 用户故事（User Stories）
一份长而详尽的编号列表，覆盖功能的各个方面。每条格式：
1. 作为 <角色>，我想要 <功能>，以便 <收益>
（示例：作为手机银行用户，我想看到各账户余额，以便更明智地决定如何花钱）

## 实现决策（Implementation Decisions）
已敲定的实现决策，可包含：将构建/修改的模块、被改动的模块接口、来自开发者的技术澄清、架构决策、schema 变更、API 契约、关键交互。
不要写具体文件路径或代码片段——它们很快就会过时。
例外：若原型产出了比文字更精确编码了某项决策的片段（状态机、reducer、schema、类型形状），可内联进相关决策并注明它来自原型；只留决策相关的精华，不是可运行的 demo。

## 测试决策（Testing Decisions）
已敲定的测试决策，包含：什么是好测试（只测外部行为，不测实现细节）、将测试哪些模块（对应 ② 划定的 seams）、代码库里同类测试的参照（prior art）。

## 范围之外（Out of Scope）
明确不在本 PRD 范围内的事。

## 其他说明（Further Notes）
关于该功能的其他补充。
```

写作要领：

- **从用户视角写问题与方案** —— 这两段讲"用户的痛和爽"，不是技术实现。
- **用户故事要长要全** —— 宁可详尽到啰嗦，覆盖正常流、边界、异常，别只写主干。
- **实现决策防过时** —— 默认不写文件路径和代码片段，只写"决策"本身；唯一例外是原型产出的、比文字更精确的决策性片段。
- **沿用项目术语** —— 全程用代码库既有的领域词汇，别另造一套叫法。

写完告知用户文件路径，交付即结束（本 skill 不接着实现该 PRD）。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 对话几乎没有可综合的内容 | ① 判定信息严重不足、且非小范围补齐 | 告诉用户当前还没足够讨论支撑一份 PRD，建议先用 jiuqing-idea-grill 把想法理清再来 |
| 无代码库可探查 | 当前目录非代码项目 | 跳过 ② 的探查与 seam 划定，PRD 中测试决策标注"待补充（无现有代码参照）"，其余照常 |
| docs/ 目录不可写 | 写文件权限错误 | 告知用户检查目录权限，或请用户指定其他输出路径 |
| 用户对划定的 seam 有异议 | ② 确认时用户说"不该测在这层" | 以用户意见为准重新划定，不强推自己的接缝方案 |
| 用户信息自相矛盾 | ① 或 ③ 中发现对话前后说法冲突（如"只做退款"和"也要做退货"） | 把矛盾点明确列出，让用户裁决以哪个为准，不擅自选择或合并 |
| PRD 模板不适用 | 功能类型特殊（如基础设施迁移、纯算法优化），七段模板的某些段落无意义 | 跳过无意义的段落，在"其他说明"中标注"因功能类型特殊，XX 段落不适用"，不硬塞内容凑格式 |
| 用户中途改主意 | 写到一半用户说"方案变了"或"范围要改" | 停下当前工作，确认新方案/新范围后再重新开始，不继续用旧信息写 PRD |
| 用户要求写代码 | h2 类场景：用户说"顺便把代码也写了" | 产出 PRD 后明确拒绝，说明本 skill 只负责需求文档，不负责实现 |
