# jiuqing-context-sync

读项目文件，提取需要消除歧义的领域术语，在项目根目录创建或增量更新一份 `CONTEXT.md` 术语表。

它解决的是「同一个词，不同人理解不一样、或代码和说法对不上」：项目里散落着各种概念（account 到底指用户还是客户？代码叫 `voidOrder` 但大家说"取消订单"），不统一就会持续误解。`CONTEXT.md` 把这些钉死成一份**纯术语表**，作为项目的领域语言基准。

它的本质是**对齐理解**，不是堆文档。所以纳入标准围绕"歧义性"而非"重要性"——一看就懂的词不收，只收会让人误解的。这样术语表保持最小且必要，不啰嗦。

## ① 读项目、提取候选术语

扫项目文件，找出符合下面纳入标准的术语。能从代码/文档可靠读到的就自己读，不要拿去问用户：

- README、docs/ —— 项目想做什么、用了哪些专有概念。
- 核心代码的命名 —— 实体、状态、关键函数名，看代码叫法和领域口语是否一致。
- 已有 `CONTEXT.md`（若存在）—— 先读它，理解现有术语，本次只做增量。

**纳入标准（满足任一即收）：**

- **有歧义 / 过载** —— 同一个词在不同语境指不同东西（account 指 Customer 还是 User）。
- **领域专有概念** —— 项目特有、外人看不懂、需要解释边界的（如"结算周期""冻结余额"）。
- **代码与口语叫法不一致** —— 用户说"取消订单"，代码里叫 `voidOrder`，这种映射必须记。
- **多处复用且边界要紧**的核心实体或关系。

**排除标准（满足任一即不收）：**

- 一看就懂、无歧义的通用词（button、id、list、save）。
- 纯技术实现名词（Redis、JWT、middleware）—— 这是实现细节，不是领域语言。
- 临时的、一次性的、不会复用的词。

判定口诀：**"两个人可能理解不一样、或代码和说法对不上"才收；一看就懂的不收。**

## ② 创建或增量更新 CONTEXT.md

写到项目根目录 `CONTEXT.md`。**懒创建**：不存在则新建，存在则只做增量，不重写整份。

条目格式，一行一条，扁平排列（不分组）：

```markdown
# CONTEXT

本文件是项目的领域术语表，统一团队对核心概念的理解。只放术语，不放实现细节。

**术语** — 从领域视角的定义。（必要时附：代码中称 `xxx`）

示例：
**Customer** — 拥有一个或多个账户的真实付费方，与登录系统的 User 不是一回事。
**冻结余额** — 账户中已被预扣、暂时不可动用的金额。（代码中称 `heldBalance`）
```

**更新策略：**

- **增** —— 出现符合纳入标准的新术语，追加。
- **改** —— 已有条目定义不准或过时，修正定义本身（改写那一条，不是再叠一条）。
- **删** —— **默认不删**。术语表是项目记忆，删条目会丢失历史理解。某概念已彻底从代码和领域移除时，标注"（已废弃）"而非物理删除。
- **冲突** —— 新用法与已有定义矛盾时，**不静默覆盖**。把冲突摆给用户裁决，例如："术语表里 cancellation 是整单取消，但代码里出现了部分取消，哪个对？" 用户裁决后再更新。之所以不擅自改，是因为 `CONTEXT.md` 跨会话长期演进，静默改写会无声腐蚀它。

## 红线：CONTEXT.md 只放术语，别无其他

`CONTEXT.md` 是纯术语表。守住这条红线，否则它会退化成大杂烩、失去作为理解基准的价值：

- ❌ 不放实现细节（用了什么库、什么数据结构、什么算法）。
- ❌ 不放架构 / 技术决策（那是 PRD 等文档的事）。
- ❌ 不当 spec、不当草稿纸、不当 TODO 清单。
- ❌ 不放具体文件路径或代码片段（易过时）。
- ✅ 只放：术语 → 领域视角的定义，必要时附"代码里叫什么"的映射。

## 收尾

更新后告知用户 `CONTEXT.md` 路径、本次新增/修改了哪些条目、有没有待裁决的冲突。交付即结束（本 skill 只维护术语表，不改其他项目文件）。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 无项目可读 | 当前目录非代码项目、无 README 无源码 | 告知用户没有可提取术语的内容，请用户指定项目或先有些代码/文档再来 |
| 提不出合格术语 | 扫完没有符合纳入标准的词（项目太通用/太早期） | 如实告知"未发现需要消歧的领域术语"，不为了凑数硬塞通用词 |
| 根目录不可写 | 写 CONTEXT.md 权限错误 | 告知用户检查目录权限，或请用户指定其他输出路径 |
| 用户对某术语定义有异议 | 用户说"这个词不是这意思" | 以用户为准修正该条，不强推自己的定义 |
