# jiuqing-agents-inject

把一套经过验证的通用 agent 行为规范，增量写入当前代码项目的 `AGENTS.md`，让该项目里的 agent 行为更可预测、更少犯错。规则模板是固定的（见同目录 `RULES.md`），本 skill 负责把它**安全地合并**进目标文件，不破坏用户已有内容。

这是固定流程，不需要执行前对齐，按下面步骤直接执行。

## 为什么只写 AGENTS.md

`AGENTS.md` 是跨平台（Codex、Claude Code、OpenCode、Gemini CLI 等）的事实标准 agent 指令文件。只写它一处，避免在 `CLAUDE.md` / `GEMINI.md` 里重复维护同一套规则。若项目用的是别名文件，建议用户用软链指向 `AGENTS.md`，而不是复制内容。

## 执行步骤

### 1. 定位目标文件

在项目根目录找 `AGENTS.md`。

- **存在** → 进入「增量合并」。先用 Read 完整读取，确认现有结构，再动手。
- **不存在** → 进入「新建」。不依赖 `/init` 之类的 CLI 内置命令——它跨平台不统一、产出的是项目概览而非行为规则，二者是不同的事。若用户想要完整项目上下文（架构、构建命令），提示他自行运行 `/init`，与本 skill 各管各的。

### 2. 读取规则模板

读同目录的 `RULES.md`，它是要写入的完整英文行为规范模板（核心原则 / 文件编辑护栏 / 硬性禁止 / 汇报标准 / 沟通 / 子任务委派格式 / 输出前自检 / 工具使用 / 任务类型聚焦表）。**原样写入，不要改写或删减条目**——这套规则的价值在于稳定一致。

### 3a. 新建（项目无 AGENTS.md）

创建一个极简 `AGENTS.md`：

```markdown
# AGENTS.md

<!-- 一行项目说明：从 README 或目录结构推断，拿不准就留一句中性描述 -->

## Behavioral Rules

<RULES.md 全文>
```

项目说明只写一行，能从 README/目录可靠推断就写，推断不出就写中性占位让用户补，**不要为了填充而编造项目细节**。

### 3b. 增量合并（项目已有 AGENTS.md）

目标是把 `## Behavioral Rules` 段补进去，同时**完全保留**用户已有内容：

- **没有行为规范段** → 用 Edit 在文件末尾追加完整的 `## Behavioral Rules` 段。
- **已有同类段**（如「行为规范」「Behavioral Rules」「Coding Guidelines」等）→ **逐小节比对**：模板里有、用户没有的小节才追加；用户已写过的同义小节**保留用户版本**，不覆盖、不重排。把你做的合并动作（追加了哪些小节、保留了哪些）如实告诉用户。
- 拿不准某个小节算不算重复时，**问用户**，不要擅自合并或丢弃。

绝不删改与本次注入无关的任何行。

### 4. 汇报

写完后用一两句话说清：目标文件路径、是新建还是合并、追加了哪些小节、保留了用户的哪些原有内容。让用户能直接 `git diff` 验证。

## 边界

- 只动 `AGENTS.md` 一个文件。要碰 `CLAUDE.md`/`GEMINI.md`/其他文件，先问。
- 不顺手改项目里别的东西，不"顺便"重排用户已有规则。
- 规则模板按需演进时改 `RULES.md`（单一信息源），不要在 SKILL.md 里另抄一份。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| RULES.md 不存在或不可读 | 文件缺失 | 告诉用户模板文件丢失，需要恢复 skills 仓库后重试；不自行编造规则 |
| 用户要求添加违反模板的规则 | 如"允许自动 commit" | 解释模板中有对应禁止条目，建议用户修改 RULES.md 模板后再注入；不绕过模板 |
| AGENTS.md 已有完整行为规范段 | 全部小节已存在 | 告诉用户无需注入，当前规范已完整 |
