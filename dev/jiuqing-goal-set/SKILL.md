# jiuqing-goal-set

扫描分析一个已有项目的现状，反推出一个**聚焦"打磨收尾、逼近发布上线"的明确目标**，产出一段可直接喂给另一个 agent 的目标设定提示词。

它解决的是「项目在手，但下一步目标不清」：项目已经能跑、有了雏形，但离能上线还差一段，差在哪、该先做什么不清楚。本 skill 通过深度扫描识别上线差距，把"该往哪走"写成一段结构化提示词交付给用户——**本 skill 只产出提示词，不执行该目标、不改项目任何文件**。

与相邻能力的边界：识别已知任务并包装成 prompt 是另一类事（那是任务已明确、只需结构化）；全自动循环迭代把项目跑到可交付是又一类事（那是自己闭环干活、产出代码）。本 skill 的独特动作是**从项目现状反推出目标**，且产出物是提示词而非代码。

## ① 深度扫描项目

目的是摸清"现在到哪了、离上线还差什么"。按下面几层从外到内扫，能从文件/代码可靠读到的就自己读，不要拿去问用户：

- **意图层** —— README、ROADMAP、CHANGELOG、docs/，搞清项目想做成什么、声称完成到哪。
- **结构层** —— 目录结构、包管理文件（package.json / pyproject.toml / Cargo.toml / go.mod 等）、入口文件，判断项目类型与规模。
- **完成度层** —— 代码里的 TODO / FIXME / `throw new Error("not implemented")` / 空函数 / 被注释掉的功能，定位未完成的功能。
- **质量层** —— 读关键模块代码，看测试覆盖（有没有测试、跑不跑得过）、CI 配置、依赖是否陈旧或有告警、错误处理与边界是否缺失。
- **上线差距层** —— 综合上面几层，列出"离能发布上线还缺的硬项"：核心功能缺口、构建/部署能不能跑通、关键 bug、文档与配置缺失、安全敏感点。

扫完用几句话向用户复述你对项目现状的判断（做到哪了、最显眼的上线差距是什么），作为对齐的输入。**不要为了凑结论编造没扫到的事实**——读不到就说读不到。

## ② 执行前对齐

本 skill 在 `depends` 中声明 `jiuqing-idea-grill`。基于上一步的扫描判断，在产出提示词**之前**先走一遍对齐流程：一次只问一个问题，每个问题带上你基于扫描得出的推荐答案。

对齐要点：

- **什么算"上线"** —— 这个项目的"发布上线/可交付"标准具体是什么？（MVP 能跑？内部可用？正式对外？）这决定目标的终点画在哪。
- **上线范围** —— 哪些差距必须在这一轮目标里补齐，哪些明确推迟到上线之后？
- **优先级** —— 多个差距里，先攻哪个？（通常先攻挡住上线的硬项，而非锦上添花）
- **约束** —— 有没有不能动的部分、必须保留的行为、时间/技术约束？

对齐后用一两句话复述共识，再进入产出。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 用户未指定项目 | 扫描目标为空或当前目录非代码项目 | 追问用户要分析哪个项目，不自行猜测 |
| 项目过于早期（无 README、无包管理文件） | 结构层扫描无信号 | 告知用户项目尚早，建议先用 jiuqing-idea-grill 理清方向再定目标 |
| 扫描结果与用户认知冲突 | 用户说"不对，这个项目已经能跑了" | 以用户信息为准，重新扫描确认 |

## ③ 产出目标提示词

用**上线导向框架**把对齐后的目标写成一段提示词，交付给用户。框架五块，按顺序组织：

1. **现状诊断** —— 项目当前到哪了（基于扫描），最关键的上线差距是什么。列出 2-3 个事实支撑判断（如"README 存在但 CHANGELOG 缺失""核心功能 X 已实现但 Y 还是 stub"），不写空泛评价。
2. **上线差距** —— 离"上线"还缺的硬项清单，每条差距写成"缺 X"或"Y 还没做"的具体陈述，已和用户对齐过范围。
3. **优先级** —— 这些差距的攻克顺序，先做什么后做什么，给出理由（如"先补测试再加功能，因为没测试的加功能是盲改"）。
4. **明确 Goal** —— 这一轮要达成的、具体可验收的目标，写成"完成 X + 验证 Y 通过"的形式，聚焦"逼近发布"，不发散。
5. **验收标准** —— 逐条列出可检验的完成条件，每条写成"X 可以 Y"或"运行 Z 命令输出 W"的形式（如"npm test 全绿""核心流程：注册→登录→创建项目→导出，端到端跑通"）。

写作要领：

- **目标具体、可验收** —— 避免"优化项目""提升质量"这类无法判断完成的空目标；写成"补齐 X 功能并让 Y 流程端到端可用"。
- **聚焦上线，不发散** —— 只写挡在发布路上的事；锦上添花、远期想法放进"上线之后"另列，不混进本轮 goal。
- **提示词自包含** —— 产出的提示词要能被另一个 agent 独立读懂，把现状背景写进去，不假设对方看过本次对话。

产出后按如下格式呈现，交付即结束（本 skill 不接着执行该目标）：

```
📋 项目现状已分析，以下是聚焦上线的目标提示词：

[完整目标提示词内容]

---
你可以把这段提示词喂给另一个 agent 去执行。如需调整目标范围或优先级请指出。
```

## ④ 自检

交付前过一遍：现状诊断是否基于真实扫描而非编造；上线差距与优先级是否已和用户对齐；Goal 是否具体可验收、聚焦上线未发散；提示词是否自包含、能被另一个 agent 独立读懂；全程是否只产出提示词、没擅自改项目文件。
