# jiuqing-skill-create

在本仓库里造一个新的 skill。先判断该不该做、再对齐需求、再起草、最后自检，全程遵循仓库的 `jiuqing` 命名与 SKILL.md 格式约定。

## ① 前置 Triage（动手前先问自己）

起草任何东西之前，先过这三道闸门。任何一道判定"不该做"，就停下来跟用户说清楚，而不是硬造。

1. **这要不要做成 skill？** —— 一次性任务、一句话 prompt 就够、agent 原生能力直接能干、或更适合写成固定脚本的，都不值得做成 skill。skill 是给会被**反复复用**的能力准备的（至少 3 个具体用例才值得抽象）。
2. **有没有现成类似的？** —— 扫一遍 `core/`、`dev/`、`content/` 已有的 skill。如果已有覆盖，改进它而不是新建一个重复的。
3. **范围是不是太大？** —— 一个 skill 只做一件事。如果用户的需求横跨多个不相关动作，建议拆成多个 skill，逐个做。

## ② 执行前对齐

本 skill 在 `depends` 中声明 `jiuqing-idea-grill`。通过 triage 后，先走一遍对齐流程，用下面这份**对齐要点**清单逐条问清楚，达成共识再起草。一次只问一个问题，每个问题带上你的推荐答案，能查上下文就别问。

对齐要点：

- **做什么** —— 这个 skill 要让 agent 具备什么能力？产出是什么？
- **何时触发** —— 用户会用什么话/什么场景触发它？（这决定 description 怎么写）
- **输入输出** —— 输入是什么形态？期望输出长什么样？有没有固定格式？
- **不该做什么** —— 边界在哪？哪些相邻场景**不**应该触发它？
- **领域与依赖** —— 属于 core / dev / content 哪类？要不要 `depends` 引用某个 core 基础 skill？
- **要不要捆绑资源** —— 有没有确定性的重复操作值得写成脚本，或大段参考资料值得拆成单独文件？

对齐后用一两句话复述共识，再进入起草。

## ③ 起草 SKILL.md

按仓库约定填这几块：

- **name** —— `jiuqing-名词-动词`，固定前缀 `jiuqing`，连字符连接，全小写，不用缩写（如 `jiuqing-image-generate`）。
- **description** —— **只写触发条件**，从"当用户……时使用"切入，描述用户会说的话和场景。**不要在描述里概括流程或步骤**：描述若总结了正文流程，agent 容易照着描述走、跳过正文。
- **category** —— `core | dev | content` 之一。
- **depends** —— 引用的基础 skill 列表，没有就 `[]`。若声明了依赖，正文里要显式说明在何处调用了它。
- **正文** —— agent 执行的行为规范，全中文，用祈使语气。

写作要领：

- **解释为什么** —— 多讲清楚"为什么要这么做"，少用一连串全大写的 MUST/NEVER。模型有理解力，给它原因比给它命令更可靠。
- **精简分层** —— 正文保持精简（理想 <100 行）。所有路径都需要的内容内联；只有部分场景才用到的大段参考，拆成同目录单独文件，并在正文写清楚"何时去读它"。
- **不硬编码平台** —— 不写死某个平台特定的路径或工具。
- **给具体例子** —— 一个好例子胜过空泛描述，但不要堆多语言/多份雷同例子。

目录结构：

```
skill-name/
├── SKILL.md          # 必须：frontmatter + 正文
├── REFERENCE.md      # 可选：大段参考资料（正文超长或领域可分时）
└── scripts/          # 可选：确定性的重复操作
```

## ④ 起草自检

交付前对照这份清单逐条过一遍，不通过就改。每条用**具体操作**验证，不靠印象判断：

- [ ] **名称合约定** —— 运行 `ls core/ dev/ content/ | grep <名称>` 确认无重名
- [ ] **描述只写触发条件** —— 检查 description 是否包含"当用户……时使用"，是否**不含**任何步骤/流程描述
- [ ] **frontmatter 完整** —— 四项齐全，用 `head -7 SKILL.md` 逐字段核对
- [ ] **depends 已说明** —— 正文搜索依赖 skill 的名称，确认有显式调用说明
- [ ] **无占位与含糊** —— 搜索 `TODO`、`FIXME`、`……`、`待定`、`视情况`，零命中才算过
- [ ] **不自相矛盾** —— 逐段读，检查同一概念是否有两种不同说法
- [ ] **无范围蔓延** —— 对照步骤 ② 对齐时定下的"做什么"，逐条检查正文是否只做了那一件事
- [ ] **正文精简** —— 统计行数，接近 100 行就拆分
- [ ] **无时效性信息** —— 搜索具体日期/版本号（如 `2024-`、`v1.2`），零命中
- [ ] **用户能验证** —— 让自己扮演用户，看这份 SKILL.md 能否指导自己完成该 skill 的全部流程

自检通过后，把成品交给用户，并说清楚它放在哪、怎么触发。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 用户需求含危险操作 | 如"执行任意 shell 命令""删除所有文件" | triage 第 1 道闸门拦截：这不该做成 skill，拒绝并解释安全风险 |
| 命名与已有 skill 冲突 | 扫描发现同名 skill | triage 第 2 道闸门拦截：建议改进已有 skill 而非新建 |
| 需求范围过大 | 需求横跨多个不相关动作 | triage 第 3 道闸门拦截：建议拆分为多个 skill |
| 对齐后用户长时间不回复 | 步骤 ② 中途停滞 | 保留已确认的部分，用默认值填充未确认项，复述后交付 |

## 安全约束

本 skill **只创建 SKILL.md 及其附属文件**（REFERENCE.md、scripts/），不执行其他操作。具体禁止：
- 不执行 `rm`、`git commit`、`git push`、`curl` 或其他与创建 skill 无关的命令
- 不在 skill 中嵌入可执行任意 shell 命令的内容（如 `system("...")`、`subprocess` 调用外部脚本）
- 不创建包含密钥、token、密码等敏感信息的 skill
- 创建的 skill 须符合仓库命名约定（`jiuqing-名词-动词`），不创建绕过命名规范的 skill
