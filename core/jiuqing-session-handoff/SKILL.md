# jiuqing-session-handoff

把当前会话压缩成一份交接文档，让一个全新的会话（可能换了模型或平台）能无损接力，不必重读整段历史。文档存到项目内的 `handoff/` 目录，纯本地、不进版本控制、不用于人与人之间交接。

## 何时用 / 何时不用

- **用**：切换执行体（换模型——额度满、换旗舰模型攻坚；换 agent 平台——如 Claude Code 转 Codex，会话不互通）、上下文窗口快爆了、当前会话被一堆失败尝试带偏想重开、探索/规划阶段结束准备进入干净的执行阶段。
- **不用**：任务已经做完没有后续（无需交接）；要把工作交给另一个人或另一台机器（本目录是本地的，clone/同事看不到，这种情况该贴 PR/issue）。

## 执行步骤

每一步都带可检验的完成标准，标准没达到不要进下一步。

1. **确认落点与意图** —— 交接文档写到仓库根的 `handoff/` 目录（不存在就建）。文件名带时间戳或主题，避免覆盖旧文档，如 `handoff/2026-06-20-auth-refactor.md`。若用户传了参数，把它当作"下一个会话的焦点"，整篇按这个焦点裁剪——只写对那个焦点有用的上下文。
   - 完成标准：明确了文件路径，且知道这次交接服务于什么焦点。

2. **梳理可接力的上下文** —— 站在"一个什么都不知道的新会话"的角度，想清楚它接手要知道什么。至少覆盖：在做什么任务、当前进展到哪、卡在哪里/下一步该干什么、关键决策及其原因、踩过的坑（哪些路走不通，省得新会话重蹈）。
   - 完成标准：新会话只读这份文档就能接着干，不必追问"前面发生了什么"。

3. **不重复已有产物** —— PRD、计划、ADR、issue、commit、diff 里已经写清楚的内容，用路径或 URL 引用，不要复制粘贴进来。交接文档只补"这些产物之外、只存在于本次对话里"的上下文。
   - 完成标准：文档里没有可被一句引用替代的大段搬运内容。

4. **写 suggested skills 段** —— 在文档里列一段"建议下一个会话调用的 skill"，让接力方拿到就知道下一步该用哪些能力，而不只是拿到一堆信息。
   - 完成标准：文档含明确的 suggested skills 段。

5. **脱敏** —— 写入前过一遍，抹掉 API key、token、密码、个人身份信息等敏感内容。
   - 完成标准：文档不含任何敏感凭据或隐私信息。

6. **写入并回报绝对路径** —— 把文档写到 `handoff/` 下，然后**把绝对路径明确输出给用户**。交接文档的全部价值在于被下一个会话读取，换了平台/模型的接力方拿不到路径就等于这份文档不存在。
   - 完成标准：用户拿到了可直接复制给新会话的绝对路径。

## 文档结构参考

不必拘泥，按本次焦点裁剪，但通常包含：

- **任务目标** —— 在做什么，要达成什么。
- **当前进展** —— 已完成 / 进行中 / 未开始。
- **下一步** —— 接力方应该先做什么。
- **关键决策** —— 定了什么、为什么这么定。
- **踩过的坑** —— 哪些路走不通，别重试。
- **相关产物** —— 用路径/URL 引用的 PRD、计划、commit、文件等。
- **suggested skills** —— 建议下一个会话调用的 skill。

## 安全约束

- 只创建 / 写入 `handoff/` 目录下的交接文档，不动其他文件。
- 不执行 `git commit`、`git push`、`rm` 等与生成交接文档无关的命令。
- 写入前必须完成脱敏，不把密钥、token、密码、隐私信息留在文档里。
