# jiuqing-idea-grill

在动手之前，先把方案、需求、意图逐个问清楚，与用户达成共识再继续。同一套提问框架支持两种入口。

## 何时生效

- **用户直接调用** —— 用户想对齐某个方案/意图/需求/计划时直接触发，对齐完成后即结束（无上层任务可交回）。
- **被其他 skill 依赖** —— 上层 skill 在 `depends` 中声明 `jiuqing-idea-grill` 时，agent 在执行该 skill 的实际任务前先走一遍本流程，对齐后交回上层 skill 继续执行。
- 对被依赖场景，`depends` 就是开关：声明即对齐，不声明即跳过；固定脚本类 skill 通常不声明依赖，因而直接执行。

## 提问框架

1. **逐个走设计分支** —— 沿着决策树一个分支一个分支地问，按依赖顺序解决（先定的决策会影响后面要问什么）。
2. **一次只问一个问题** —— 等用户回答后再问下一个。一次抛多个问题会让人无所适从。
3. **每个问题都给推荐答案** —— 不要光问，要带上你的建议和理由，让用户能快速确认或推翻。
4. **能查代码/上下文就别问** —— 凡是可以通过查代码、读文件、看上下文得到答案的，自己去查，不要拿去问用户。

## 对齐要点（可选）

引用本 skill 的上层 skill 可在自己的正文里提供一个"对齐要点"清单，列出该领域执行前必须确认的关键项（例如生图必问风格、比例、用途）。本 skill 负责把这些要点逐条问出来。

若上层 skill 没有提供清单，则完全依据上述通用提问框架，结合当前任务现场生成问题。

## 退出与交回

- 当关键决策点都已对齐——用户已确认，或答案可从代码/上下文可靠推断——即停止提问，不要无限追问。
- 退出时用一两句话复述达成的共识。
- 若是被上层 skill 依赖触发，随后将控制权交回上层 skill，继续执行它的实际任务；若是用户直接调用，复述共识后即结束。

## 失败路径

| 场景 | 信号 | 处理 |
|------|------|------|
| 用户说"别问了直接做" | 明确拒绝继续对齐 | 解释为什么需要先对齐（避免做错方向），如用户仍坚持则记录已确认的部分，将未确认项用默认值填充，复述后交回 |
| 输入极度模糊 | 上下文为空且用户给不出方向 | 不要自行编造方案主题，告诉用户需要明确方向才能对齐 |
| 用户中途切换需求 | 对齐过程中突然改方向 | 用新方向重新开始对齐分支，之前已确认的决策作废 |
